# Code review — confirmed bugs

## 1. [medium] dota/win32.ts — Scroll-down throws: [uint32] cast of negative wheel delta aborts the whole spec
The scroll action computes the wheel delta as `[uint32]([int]$a.amount * 120)`. mouse_event's `data` is a DWORD that carries a SIGNED wheel delta (negative = scroll down). The control.tools schema explicitly documents amount as "+ up / - down", so negative values are a supported, reachable input. But casting a negative .NET value to [uint32] throws OverflowException ("Value was either too large or too small for a UInt32"). With `$ErrorActionPreference='Stop'` this terminating error aborts the entire PowerShell run mid-sequence, so no JSON is emitted and runWin32Spec returns the opaque `No result from window engine` error. I reproduced: `[uint32]([int](-3) * 120)` => throws on the actual PS 5.1 in this env. Scroll-up (positive) works; scroll-down is completely broken.

**Fix:** Convert the signed delta to a DWORD via an unchecked/2's-complement conversion instead of a checked [uint32] cast, e.g.: `$delta=[int]$a.amount*120; $data=[System.BitConverter]::ToUInt32([System.BitConverter]::GetBytes([int]$delta),0); [W32]::mouse_event($WHEEL,0,0,$data,[UIntPtr]::Zero)`. (BitConverter approach verified to yield 4294966936 for -360.)

## 2. [med] dota/win32.ts — Spec JSON read with wrong encoding corrupts non-ASCII `text` actions
The spec file is written by Node as BOM-less UTF-8 (`writeFile(specPath, JSON.stringify(spec), "utf8")`), but the PS engine reads it with `Get-Content -Raw -Path $SpecPath` WITHOUT `-Encoding UTF8`. In Windows PowerShell 5.1 (the edition in this env) Get-Content with no -Encoding decodes a BOM-less file using the system ANSI codepage, not UTF-8. I reproduced: a UTF-8 spec containing `日本語` is read back as mojibake `ж—Ґжњ¬иЄћ` under default encoding, but correctly with `-Encoding UTF8`. Any `text` (or `key`) action containing non-ASCII (accented/localized names, em-dashes, CJK) is silently corrupted before being sent to the game via SendKeys. Pure-ASCII payloads are unaffected, which is why it slips through.

**Fix:** Read as UTF-8 explicitly: `$spec = Get-Content -Raw -Path $SpecPath -Encoding UTF8 | ConvertFrom-Json` (or read via `[System.IO.File]::ReadAllText($SpecPath,[Text.Encoding]::UTF8)`). The same applies to capture.ts's PS_SCRIPT if it ever read a spec, but it does not — only win32.ts is affected here.

## 3. [low] dota/panorama-decompile.ts — decompilePanorama picks the single longest printable run — can grab the wrong block and emit garbage/truncated source
The decompiler scans for printable runs (>=16 chars) and keeps only the single longest one. Two correctness failures: (1) The metadata-skip filter only fires when r.length < 800: `if (/m_(Input|Compiler|...)|RED2|DATA\b|VructResource/.test(r) && r.length < 800) continue;`. A long compiler/dependency block (>=800 chars) is therefore NOT skipped, and since 'longest wins', it can beat the real source and be returned as the file body — pure garbage. (2) Real source is split into multiple printable runs whenever a single non-printable byte appears mid-source (compiled CSS/JS frequently embeds NULs or >0x7E bytes for tokens/offsets). Only the longest fragment is kept, silently dropping the rest, so the stored .css/.js is a truncated partial. (3) For .vcss_c/.vjs_c there is no trailing-trim (only .vxml_c gets the `<` head-trim), so trailing compiler metadata that is contiguous with the source tail is kept appended to the recovered text.

**Fix:** Don't rely solely on 'longest run'. Detect the source by content shape per extension: for .vcss_c pick the run containing CSS markers (e.g. `{`/`}` and `:` density or `@keyframes`); for .vjs_c the run with JS markers (`function`/`var`/`=>`/`(function`); for .vxml_c the run starting with `<root`/`<`. Among candidate runs, score by ratio of source-like chars rather than raw length, and apply the metadata-skip regex unconditionally (drop the `&& r.length < 800`). Also trim trailing metadata for css/js (e.g. cut at the last `}` for css, and strip a trailing run that matches the metadata regex). Consider concatenating adjacent source runs rather than keeping only one.

## 4. [low] dota/reflib.ts — resolveVpk picks an arbitrary .vpk when no <id>.vpk exists, including non-dir VPK split archives
When the preferred `<id>.vpk` is absent, resolveVpk returns `files[0]` of all `*.vpk` in the dir. readdir order is not guaranteed and the dir may contain split archives like `pak01_000.vpk` / `pak01_dir.vpk`. Vpk.open requires the *_dir.vpk (it derives split-archive paths via `_dir.vpk -> _NNN.vpk`). Picking a non-_dir part (e.g. `..._001.vpk`) will either fail the magic check or read a tree-less data archive, causing ingest to throw or extract nothing. The selection should prefer a *_dir.vpk over arbitrary parts.

**Fix:** Filter/sort to prefer a directory VPK: pick a file matching /_dir\.vpk$/i first, else the lone .vpk; explicitly exclude split parts matching /_\d{3}\.vpk$/i unless it's the only candidate. e.g. `const dirVpk = files.find(f=>/_dir\.vpk$/i.test(f)) ?? files.find(f=>!/_\d{3}\.vpk$/i.test(f)) ?? files[0];`

## 5. [medium] c:\Users\work\dota2-workshop-mcp\src\dota\audit.ts — Precache regex \bPrecache\b never matches the actual precache API calls (false 'precache-missing')
Line 70 uses `!/\bPrecache\b/.test(serverBlob)`. The trailing word boundary `\b` requires a non-word char immediately after 'Precache', but every real precache CALL is `PrecacheResource`, `PrecacheModel`, `PrecacheUnitByNameSync`, `PrecacheParticle`, `PrecacheEverythingFromTable`, etc. — all continue with word chars, so `\bPrecache\b` does NOT match them. Verified: `/\bPrecache\b/.test('PrecacheResource("x")')` === false, same for PrecacheModel / PrecacheUnitByNameSync. The regex only matches a literal bare word `Precache` (e.g. the `function Precache(context)` declaration). So an addon that correctly precaches via PrecacheResource/PrecacheUnitByNameSync but does not happen to also write the standalone word 'Precache' (common in TS-to-Lua output, or when precache logic lives outside a function named exactly Precache) gets a false 'precache-missing' warning; conversely the suggestion text 'No Precache() found' is misleading.

**Fix:** Drop the trailing boundary so it matches the call family, e.g. `!/\bPrecache\w*\(/.test(serverBlob)` (matches Precache(, PrecacheResource(, PrecacheModel(, PrecacheUnitByNameSync(, …). Optionally test for the specific known functions: `/\bPrecache(Resource|Model|Unit\w*|Particle\w*|EverythingFromTable)?\s*\(/`.

## 6. [low] c:\Users\work\dota2-workshop-mcp\src\dota\audit.ts — Tooltip presence test uses substring includes() — prefix collisions cause missed 'missing-tooltip' (false negatives)
Line 170 checks `locTokens.includes(token)` where token is `DOTA_Tooltip_ability_<key>`. Because abilities/items frequently share a prefix (e.g. `gemtd_remove` vs `gemtd_remove_all`, `gemtd_choose_stone` vs `gemtd_choose_update_stone`), an ability with NO tooltip whose name is a prefix of another ability that DOES have a tooltip will be silently treated as having a tooltip. Verified: a locale containing `DOTA_Tooltip_ability_gemtd_remove_all` makes `includes('DOTA_Tooltip_ability_gemtd_remove')` return true, so `gemtd_remove`'s genuinely missing token is never reported.

**Fix:** Match the full token followed by a non-identifier delimiter (quote/whitespace), not a bare substring. e.g. build a per-key regex `new RegExp('DOTA_Tooltip_ability_' + escapeRegExp(key) + '(?![A-Za-z0-9_])')` and test that against locTokens, or pre-extract all token keys with `/"?(DOTA_Tooltip_ability_[A-Za-z0-9_]+)"?/g` into a Set and check `set.has(token)`.

## 7. [low] c:\Users\work\dota2-workshop-mcp\src\dota\audit.ts — Send_ServerTo\w+ regex also matches Send_ServerToAllClients and can capture the payload as the event name
Line 83's `Send_ServerTo\w+\(\s*[^,]+,\s*["']([^"']+)["']` treats the event name as the SECOND argument (correct for Send_ServerToPlayer/Send_ServerToTeam). But `\w+` also matches `Send_ServerToAllClients`, whose event name is the FIRST argument (handled separately on line 85). When an AllClients call has a string-typed second arg, line 83 captures that second string (the payload) as a bogus event name. Verified: for `Send_ServerToAllClients("ev_name", "payload")`, line 83 captures `payload` while line 85 correctly captures `ev_name`; both get added to serverSends, so `payload` becomes a phantom server-send that yields a spurious 'custom-event-no-subscriber' finding.

**Fix:** Exclude AllClients from the two-arg regex so only the dedicated line-85 regex handles it. Use a negative lookahead or restrict the suffix: `/Send_ServerTo(?:Player|Team)\w*\(\s*[^,]+,\s*["']([^"']+)["']/g` (covers Send_ServerToPlayer, Send_ServerToTeam, and variants while excluding AllClients).

## 8. [low] c:\Users\work\dota2-workshop-mcp\src\dota\audit.ts — Hidden-ability skip breaks when AbilityBehavior is a repeated/array value (and entry is an array)
blockToObject collapses repeated keys into arrays. If an ability key appears twice (duplicate definitions, or merged via tooling), `obj[key]` is an array — `typeof []==='object'` passes the line-164 guard, then `(obj[key] as Record).AbilityBehavior` is `undefined`, so the HIDDEN skip never triggers and a hidden ability may be flagged. Same if a single entry has repeated `AbilityBehavior` lines: `.AbilityBehavior` becomes an array and `String([...])` joins with commas — `/HIDDEN/i` still works there, but the array-entry case loses the behavior entirely. Not a crash, but produces incorrect missing-tooltip findings in those edge cases.

**Fix:** Guard against arrays explicitly: `const entry = obj[key]; if (key==='Version' || typeof entry!=='object' || Array.isArray(entry)) continue;` and normalize behavior via `const beh = Array.isArray(entry.AbilityBehavior) ? entry.AbilityBehavior.join(' ') : String(entry.AbilityBehavior ?? '')` before the HIDDEN test.

## 9. [low] c:\Users\work\dota2-workshop-mcp\src\dota\audit.ts — missing-tooltip message hardcodes 'addon_english.txt' and mislabels singular (says it scans english only / 'abilitie')

## 10. [medium] tools/control.tools.ts — dota_selftest sends mcp_assert WITHOUT quoteLua — multi-token/quoted/multiline asserts break and are falsely reported FAIL
In the asserts loop, the expression is sent raw: `vc.send(`mcp_assert ${expr}`)`. Compare dota_lua_eval which sends `mcp_eval ${quoteLua(code)}`. quoteLua collapses newlines to spaces and wraps the whole thing as ONE quoted console token. Without it: (1) a newline in `expr` terminates the console command (rest is dropped or run as a second command); (2) any `"` in the expression re-tokenizes so the SDK's echoed `code` (cmd_assert prints `"::", code` where code=joinArgs(args,2)) no longer contains `expr.slice(0,40)`, so waitForLine never matches and the step is reported FAIL with detail 'no response from DebugSDK' even when the assert actually PASSed. The wait matcher `l.text.includes(expr.slice(0,40))` was clearly written assuming the SDK echoes the exact expr, which only holds when the expr is sent as a single quoted token like eval does. This is an inconsistency between two tools that drive the same SDK.

**Fix:** Quote the expression exactly like dota_lua_eval: `vc.send(`mcp_assert ${quoteLua(expr)}`)` (export/share quoteLua from debugsdk.tools, or inline the same `"`+collapse-newlines+`"`+escape logic). Then the SDK echo will contain the (newline-collapsed) expr and the slice(0,40) match against the collapsed form will be reliable. Better still, match on a tighter token (e.g. the PASS/FAIL marker plus the trailing `:: <code>`) computed from the same normalized string you send.

## 11. [low] tools/debugsdk.tools.ts — quoteLua does not escape backslashes — Lua code containing '\\' is corrupted when sent
quoteLua wraps code in double quotes and only collapses newlines and converts `"`->`'`, but leaves backslashes untouched. The Source console treats backslash as an escape inside a double-quoted token, so a literal backslash in the Lua snippet (e.g. a string pattern like `('%d'):gsub('\\.', ...)` or a Windows path) is consumed/altered by the console tokenizer before it ever reaches loadstring. The result is wrong Lua or an EVAL_ERR, with no indication why. The doc comment claims it makes 'arbitrary Lua safe to pass as ONE quoted console token' which is not true for backslashes.

**Fix:** Escape backslashes before wrapping (and before any other quoting): `code.replace(/\\/g, "\\\\").replace(/\r?\n/g, " ").replace(/"/g, "'")`, or verify the exact console tokenizer escaping rules and apply them. At minimum document the backslash limitation if escaping is intentionally omitted.

## 12. [medium] tools/debug.tools.ts — dota_screenshot picks the LAST file in readdir order, not the newest — can return a stale/wrong screenshot
The detection loop sets `found = name` for every qualifying image without breaking and without comparing modification times, so `found` ends as whatever name happens to be last in the (unsorted) readdir() result. If more than one image qualifies in a pass — e.g. a pre-existing file whose mtime is within the 1s `sinceMs` window, multiple manual screenshots, or the engine writing more than one file — the tool may return an older/unrelated image instead of the just-captured frame. Pre-existing files are also accepted purely on mtime>=sinceMs, so any file touched in the last second (not necessarily the one `jpeg` produced) can be picked.

**Fix:** Track the newest candidate by mtime instead of last-seen name: stat every candidate, keep the one with the max mtimeMs (and prefer brand-new names not in `before`). e.g. accumulate `{name, mtime}` and select the maximum after the scan, or break early only on a genuinely new (not-in-before) file.

## 13. [low] tools/debugsdk.tools.ts — dota_debug_dump regex mis-parses array payloads (heroes/units) — returns raw string instead of JSON
The payload extractor `/(\{.*\}|\[.*\])\s*$/` tries the object alternative `\{.*\}` first. For the heroes/units dumps the SDK emits a JSON ARRAY like `[MCP] DUMP heroes [{...},{...}]`. Regex alternation finds the leftmost match, and the leftmost `{` is the first array element, so `\{.*\}` greedily matches `{...},{...}` (first inner `{` to last `}`), dropping the enclosing `[ ]`. That captured string `{...},{...}` is not valid JSON, JSON.parse throws, and the tool silently falls back to returning the raw console line as a string instead of structured data. So `section:'heroes'` and `section:'units'` never return parsed JSON.

**Fix:** Put the array alternative first so a top-level array is captured whole, or anchor on the structural opener: e.g. `/(\[.*\]|\{.*\})\s*$/` (array first), or better, find the first `{` or `[` and take to end-of-line: `line.text.match(/([\[{].*[\]}])\s*$/)`. Verify against both object (state) and array (heroes/units) outputs.

## 14. [low] scaffold/systems.ts — RPC comment promises HTTP yielding that coroutine.wrap can't deliver — handler that yield()s on HTTP hangs forever, no reply ever sent
rpc.lua wraps handlers in coroutine.wrap+xpcall and the header comment claims handlers 'can yield on CreateHTTPRequestScriptVM (backend bridge)'. But CreateHTTPRequestScriptVM's req:Send(cb) is callback-based and does NOT suspend the calling coroutine — there is nothing in this scaffold that resumes the wrapped coroutine after HTTP completes. A user who follows the comment and writes `local res = coroutine.yield(); ...` (or any yielding bridge) will permanently suspend that coroutine: the out.__rpc reply lines after the yield never run, so the client callback never fires (and with a client timeout it always returns {error:'timeout'}). The coroutine.wrap only adds value for xpcall crash-isolation, not for yielding on HTTP.

**Fix:** Either drop the misleading 'can yield on HTTP' claim (the wrap is only for crash isolation), or make the pattern actually async: have the handler accept a `reply` continuation callback and send the response from inside the HTTP Send callback instead of relying on coroutine.yield. Document that req:Send is callback-based, not yielding.

## 17. [low] C:\Users\work\dota2-workshop-mcp\src\tools\workshop.tools.ts — workshop_read swallows the real error and always reports "not found"
The try/catch around the read+decompile catches EVERY error and unconditionally returns `error("\"${path}\" not found in ${id}. Use workshop_inspect to list paths.")`. This conflates genuinely-missing paths (vpk.read throws "Not found in VPK") with real failures — e.g. a corrupt VPK, a UTF-8 decode problem, or a decompile bug — all of which get mislabeled as "not found". When the file actually IS present but unreadable, the user is told it doesn't exist and sent to re-run workshop_inspect (which will list it), producing a confusing loop. The catch binding is even discarded (`catch {`), so the real message is lost.

**Fix:** Capture and inspect the error: `catch (e) { const msg = e instanceof Error ? e.message : String(e); if (/Not found in VPK/i.test(msg)) return error(`"${path}" not found in ${id}. Use workshop_inspect to list paths.`); return error(msg); }` so genuine read/decompile failures surface their real cause.

## 18. [low] C:\Users\work\dota2-workshop-mcp\src\tools\workshop.tools.ts — panorama_decompile single-file mode rejects source-only panorama and mislabels the reported path

If a VPK ships an uncompiled source panorama file (e.g. plain panorama/.../hud.css with no .vcss_c), the tool returns an error claiming it is "not in {id}" even though the file is present and readable — unlike workshop_read, which falls back correctly. **Fix:** Before erroring, fall back to the raw source if present: `if (!vpk.entries.has(key)) { if (vpk.entries.has(path.toLowerCase())) { /* read source directly without decompile */ } else return error(...); }`. Also report `path` (the user input) in the json payload rather than `key`.

## 19. [low] C:\Users\work\dota2-workshop-mcp\src\tools\assets.tools.ts — assets_list with a `type` filter can return an empty file list because the walk cap is shared and exhausted by the first root
walk() stops as soon as `out.length >= cap` (5000). It is called for contentDir then gameDir into the SAME accumulator. If contentDir alone has >=5000 files, gameDir is never walked, so compiled game assets (e.g. vmdl_c/vtex_c that only live under gameDir) are silently omitted from both the byType summary and the `files` list. With a `type` filter for such an extension the user can get `files: []` even though the assets exist on disk. assets_search has the same shared-cap behaviour (cap 10000).

**Fix:** Either give each root its own budget (call walk per-root into separate arrays then merge, capping the total afterward) or raise/remove the cap when a `type` filter is supplied so the requested extension is fully represented.
